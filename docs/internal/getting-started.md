# Getting started

This page takes you from `pip install` to a generated dataset, then to your own builder config from scratch. Five minutes of reading; ten if you run every command.

## 1. Install

```bash
pip install plotsim
```

Verify:

```text
$ plotsim --version
plotsim 0.5.0
```

plotsim requires Python 3.10+. The core engine has six dependencies (`numpy`, `scipy`, `pandas`, `pyyaml`, `pydantic`, `faker`) and never makes a network call at generation time.

Optional extras:

| Extra | Install | Adds |
| ----- | ------- | ---- |
| `parquet` | `pip install plotsim[parquet]` | Parquet output via pyarrow |
| `ui` | `pip install plotsim[ui]` | Streamlit UI dependencies (V2 preview) |

To build this docs site locally, install MkDocs Material directly: `pip install "mkdocs-material>=9.5"` (not packaged as a plotsim extra).

## 2. Run a bundled template

plotsim ships with six builder templates and five engine-direct templates. List them:

```text
$ plotsim list-templates
Builder templates (recommended — front door for new users):
  bare_minimum  Smallest valid builder config — start here
  education     Builder shape: University course enrollment
  hr            Builder shape: HR engagement / attrition
  marketing     Builder shape: Marketing campaigns / attribution
  retail        Builder shape: Retail transactions / loyalty
  saas          Builder shape: B2B SaaS customer success

Engine-direct templates (full PlotsimConfig YAML):
  ...
```

Pick one and copy it out so you can edit it later:

```bash
plotsim template saas -o config.yaml
```

Preview what it will generate before committing the wall clock:

```text
$ plotsim info config.yaml
Domain: B2B SaaS customer success
Entity type: company
Entities: 95 across 6 cohort(s) (promising_client, steady_enterprise, slow_churn, seasonal_accounts, dormant, turnaround)
Time window: 2023-01 to 2024-12 (24 months)
Metrics: 6 (engagement, mrr, support_tickets, feature_adoption, churn_risk, nps)
Tables: 9 (4 dim, 3 fact, 2 event)
Seed: <auto-generated>
```

Then generate it:

```bash
plotsim run config.yaml -o ./output --validate
```

`--validate` runs the post-generation integrity checks (PKs unique, every FK resolves, date spine has no gaps) and prints the result. Without the flag, generation is the same; you just don't see the report on stdout. The full report is always written to `output/validation_report.txt`.

Same flow from Python:

```python
from plotsim import create_from_yaml, generate_tables, write_tables

cfg = create_from_yaml("config.yaml")
tables = generate_tables(cfg)
write_tables(tables, cfg)
```

`create_from_yaml()` reads the builder YAML, runs the same UserInput validators the CLI does, and returns a fully-validated `PlotsimConfig`. From there `generate_tables()` and `write_tables()` are the same surfaces the CLI uses internally.

## 3. Inspect the output

```text
output/
├── dim_date.csv                 24 rows — one per month
├── dim_company.csv              95 rows — one per company entity
├── dim_user.csv                 sub-entities under each company
├── dim_plan.csv                 reference lookup
├── fct_engagement.csv           2280 rows — 95 companies × 24 months
├── fct_revenue.csv              2280 rows
├── fct_support_tickets.csv      2280 rows
├── evt_login.csv                hundreds — driven by engagement (proportional)
├── evt_churn.csv                tens — fires when churn_risk crosses threshold
├── config.yaml                  frozen copy of the input config
└── validation_report.txt        FK + PK + spine integrity log
```

A peek at `fct_engagement.csv`:

```text
"date_key","company_id","engagement_score","feature_adoption","customer_sentiment"
20230101,"c-001","0.0802","0.0000","at_risk"
```

`engagement_score` is not a random number between 0 and 1. It's the value of the `engagement` metric (a beta distribution at the trajectory position that company `c-001` reached in January 2023). `c-001` is in the `promising_client` segment, which is assigned the `growth > spike_then_crash > flat` archetype — a sigmoid rise to peak, then a step down, then a low plateau. So `c-001` starts low, climbs through mid-2023, peaks, drops, and ends 2024 on the low plateau.

`evt_login` has variable row count per (company, period). The count is `proportional` to `engagement` × 5 — five times the engagement metric value. So when `c-001`'s engagement is 0.08 in January, the login count is small; when engagement reaches 1.0 in late 2023, login count is much higher. Same trajectory, different table.

`evt_churn` fires only when `churn_risk` stays above 0.7 for three consecutive periods. It's a *threshold* event, not a *proportional* event — same primitive, different driver.

If you join the fact and event tables in DuckDB or your warehouse, you will see logins climb with engagement and churn events appear in the months where engagement collapsed. Not because the tool injected that pattern table-by-table — because every column read from the same trajectory.

## 4. Build your first config

The smallest builder config is six fields. Save this as `tutorial.yaml`:

```yaml
about: "Tutorial — agile teams"
unit: team

window:
  start: 2024-01
  end: 2024-12
  every: monthly

metrics:
  - {name: velocity,  type: count, polarity: positive}
  - {name: bug_count, type: count, polarity: negative}

segments:
  - {name: team_alpha, count: 5, archetype: growth}
  - {name: team_beta,  count: 5, archetype: growth}
  - {name: team_gamma, count: 5, archetype: growth}
```

`about`, `unit`, `window`, `metrics`, and `segments` are the only required fields. The interpreter fills in everything else: domain name, archetypes, entity declarations, the date dim, the per-unit dim, and a single `fct_team` fact table with `velocity` and `bug_count` columns.

Validate without generating:

```bash
plotsim validate tutorial.yaml
```

Generate:

```bash
plotsim run tutorial.yaml
```

You now have `output/dim_date.csv`, `dim_team.csv`, `fct_team.csv`. `fct_team.csv` has 180 rows (15 teams × 12 months) — each team traces the same S-curve from low values to high values, scaled by the count distribution.

## 5. Add a connection between metrics

The single-archetype config is trajectory-driven but has nothing to correlate. Add a connection between the two metrics. Add this block at the bottom:

```yaml
connections:
  - velocity opposes bug_count
```

`opposes` translates to a configured Pearson coefficient of −0.55. `polarity: negative` already means `bug_count` falls when the trajectory rises (the trajectory-first invariant), so velocity-up tends to mean bugs-down. The explicit `opposes` connection tightens that relationship beyond what the trajectory alone delivers. Re-run:

```bash
plotsim run tutorial.yaml
```

The new `bug_count` column in `fct_team.csv` shows higher values when velocity is low and lower values when velocity is high. The configured connection tightens that relationship; the trajectory was already contributing in the same direction. To measure configured-correlation fidelity in isolation, use a `flat` archetype to decouple the two signals (this is how the [statistical fidelity](https://github.com/mohossam-ae/plotsim/blob/main/docs/statistical-fidelity.md) sweeps measure the ±0.10 tolerance band).

## 6. Same flow from Python kwargs

If you'd rather skip YAML entirely, `create()` accepts the same shape as kwargs:

```python
from plotsim import create, generate_tables, write_tables

cfg = create(
    about="Tutorial — agile teams",
    unit="team",
    window=("2024-01", "2024-12", "monthly"),
    metrics=[
        {"name": "velocity",  "type": "count", "polarity": "positive"},
        {"name": "bug_count", "type": "count", "polarity": "negative"},
    ],
    segments=[
        {"name": "team_alpha", "count": 5, "archetype": "growth"},
        {"name": "team_beta",  "count": 5, "archetype": "growth"},
        {"name": "team_gamma", "count": 5, "archetype": "growth"},
    ],
    connections=["velocity opposes bug_count"],
)
tables = generate_tables(cfg)
write_tables(tables, cfg)
```

Same validation, same output. Useful in notebooks where the config evolves cell-by-cell, or in pipelines where the config comes from an upstream source rather than a file.
